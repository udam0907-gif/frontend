# DONE_CRITERIA.md — 완료 기준 명세서

> 이 문서는 작업 유형별로 **"무엇이 되면 완료인가"**를 정의한다.
> verifier 에이전트는 이 기준에 따라 PASS/FAIL을 판정한다.
> 이 기준을 충족하지 못하면 어떤 경우에도 "완료"라고 보고하지 않는다.

---

## 🎯 모든 작업 공통 기준

다음 4가지는 **모든 작업에 공통**으로 적용된다:

| 항목 | 기준 |
|------|------|
| **컨테이너 상태** | `docker ps`에서 모든 컨테이너 `Up` 상태 |
| **에러 로그** | 작업 후 새로 발생한 ERROR/CRITICAL 로그 없음 |
| **보호 영역** | 보호 영역 파일이 git diff에 포함되지 않음 |
| **범위 일치** | 변경된 파일이 작업 지시서 범위와 일치 |

---

## 📋 작업 유형별 완료 기준

### 1. 백엔드 API 엔드포인트 추가/수정

#### 필수 증거
```bash
# 1. 컨테이너 정상 기동
docker ps | grep rnd_backend  # → "Up" 상태 확인

# 2. 백엔드 로그에 시작 에러 없음
docker logs rnd_backend --tail 30 | grep -iE "error|exception"
# → 결과 없거나 무관한 기존 로그만

# 3. 변경한 엔드포인트 직접 호출 성공
curl -X [METHOD] http://localhost:8000/api/v1/[엔드포인트] \
  [-H "..."] [-d "..."] \
  -w "\nHTTP %{http_code}\n"
# → 200 또는 201

# 4. 응답 JSON 구조가 의도와 일치
# (응답 본문 캡처 후 사용자가 의도와 비교)
```

#### 통과 조건
- [ ] HTTP 200/201 응답
- [ ] 응답 본문이 schema 명세와 일치
- [ ] 백엔드 로그에 새 에러 없음
- [ ] 기존 엔드포인트 회귀 테스트 통과

#### 불충분 사례 (NOT DONE)
- ❌ 코드 작성만 하고 컨테이너 재시작 안 함
- ❌ Swagger UI에서만 보고 실제 호출 안 함
- ❌ 422 떠도 "스키마는 맞췄으니 됨"으로 처리

---

### 2. 프론트엔드 페이지 추가/수정

#### 필수 증거
```bash
# 1. 빌드 성공
docker logs rnd_frontend --tail 50 | grep -iE "error|failed"
# → TypeScript 에러 없음

# 2. 페이지 응답 200
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3001/[경로]

# 3. 사용자에게 브라우저 확인 요청 (UI는 자동검증 한계)
```

#### 통과 조건
- [ ] TypeScript 컴파일 에러 없음
- [ ] 페이지 200 응답
- [ ] 콘솔(F12)에 새 에러 없음 (사용자 확인)
- [ ] 의도한 UI 변경이 화면에 반영됨 (사용자 확인)
- [ ] 다른 페이지가 깨지지 않음 (사용자 확인)

#### 불충분 사례 (NOT DONE)
- ❌ 코드 수정만 하고 핫 리로드 확인 안 함
- ❌ "TypeScript는 통과했으니 됨"
- ❌ 브라우저 확인 없이 통과

---

### 3. DB 모델 / 마이그레이션 변경

#### 필수 증거
```bash
# 1. 마이그레이션 파일 생성됨
ls backend/migrations/versions/ | tail -3

# 2. 마이그레이션 적용 성공
docker exec rnd_backend alembic current
# → 새 revision으로 표시

# 3. 테이블 구조 확인
docker exec rnd_postgres psql -U postgres -d rnd_expense_db -c "\d [테이블명]"
# → 의도한 컬럼 존재

# 4. 기존 데이터 무결성
docker exec rnd_postgres psql -U postgres -d rnd_expense_db -c "SELECT COUNT(*) FROM [테이블명];"
# → 기대 행 수와 일치
```

#### 통과 조건
- [ ] 마이그레이션 파일이 의도한 변경만 포함
- [ ] `alembic upgrade head` 성공
- [ ] 컬럼/제약조건/인덱스가 의도대로
- [ ] 기존 데이터 손실 없음
- [ ] 다운그레이드(`downgrade()`) 함수 작성됨

#### 불충분 사례 (NOT DONE)
- ❌ 모델만 수정하고 마이그레이션 안 만듦
- ❌ 마이그레이션 만들고 적용 안 함
- ❌ `--autogenerate`만 믿고 생성된 파일 검토 안 함

### 4. PDF/문서 처리 기능

#### 필수 증거
```bash
# 1. 실제 샘플 PDF로 테스트
curl -X POST http://localhost:8000/api/v1/projects/extract-pdf \
  -F "file=@샘플.pdf" \
  -w "\nHTTP %{http_code}\n"

# 2. 추출 결과 확인
# → JSON 응답의 모든 필드가 PDF 내용과 일치하는지 사용자가 확인
```

#### 통과 조건
- [ ] 샘플 PDF 업로드 성공
- [ ] 추출된 데이터가 원본과 의미적으로 일치
- [ ] 잡음 데이터 (예: `(480)` 같은 숫자)가 회사명에 들어가지 않음
- [ ] 빈 필드는 빈 값으로 (가짜 데이터 채우기 금지)
- [ ] 추출 실패 시 명확한 에러 메시지

#### 불충분 사례 (NOT DONE)
- ❌ "이론적으로 추출됨" — 실제 파일 테스트 없음
- ❌ "필드는 다 채워졌네" — 내용이 잘못됐는데 필드 존재만 확인

### 5. DOCX/XLSX 문서 생성

#### 필수 증거
```bash
# 1. 생성 API 호출
curl -X POST http://localhost:8000/api/v1/documents/generate \
  -d '{"expense_item_id": ...}' \
  -w "\nHTTP %{http_code}\n"

# 2. 생성된 파일 다운로드
curl -O http://localhost:8000/api/v1/documents/[id]/download

# 3. 파일을 사용자에게 전달하여 직접 확인
```

#### 통과 조건
- [ ] 파일이 정상적으로 생성됨 (0 byte 아님)
- [ ] 파일이 손상되지 않음 (Word/Excel에서 열림)
- [ ] **레이아웃이 깨지지 않음**
- [ ] **placeholder가 남아있지 않음** (`{{...}}` 같은 게 보이면 실패)
- [ ] 셀/필드 값이 정확히 채워짐
- [ ] 합계/계산 값이 일치

#### 불충분 사례 (NOT DONE)
- ❌ 파일은 생성됐지만 placeholder 남아있음
- ❌ 셀이 잘못된 위치에 들어감
- ❌ 사용자가 열어보지 않음

### 6. UI 디자인 변경 (CSS/레이아웃)

#### 필수 증거
- 변경 전후 스크린샷 (사용자 제공)
- 다양한 화면 크기에서 확인 (모바일/태블릿/데스크톱)

#### 통과 조건
- [ ] 의도한 시각적 변경 적용됨
- [ ] 기존 레이아웃이 깨지지 않음
- [ ] 다른 페이지에 영향 없음 (글로벌 CSS 수정 시 특히)
- [ ] 반응형 동작 정상

#### 불충분 사례 (NOT DONE)
- ❌ 코드만 수정하고 브라우저 확인 안 함
- ❌ 한 화면에서만 보고 다른 페이지 영향 확인 안 함

### 7. 버그 수정

#### 필수 증거
```bash
# 1. 버그가 발생하던 정확한 시나리오 재현
[원래 버그 발생 명령 또는 사용자 액션]
# → 이제 발생하지 않음 확인

# 2. 회귀 테스트
# → 버그 수정으로 다른 기능이 깨지지 않았는지
```

#### 통과 조건
- [ ] **원래 버그 시나리오에서 버그가 더 이상 발생하지 않음**
- [ ] 동일 영역의 다른 기능이 정상 작동
- [ ] 버그의 원인이 정확히 파악되고 문서화됨
- [ ] 같은 종류의 버그가 다른 곳에 없는지 확인

#### 불충분 사례 (NOT DONE)
- ❌ "수정한 것 같음" — 재현 시도 안 함
- ❌ 증상만 가리고 원인은 그대로
- ❌ 회귀 테스트 안 함

### 8. 리팩토링

> ⚠️ 이 프로젝트에서 리팩토링은 **사용자가 명시적으로 지시한 경우만** 수행한다.
> "하는 김에" 리팩토링은 절대 금지.

#### 필수 증거
```bash
# 모든 기존 기능이 그대로 작동하는지
# → 회귀 테스트 (관련 엔드포인트 전부 호출)
```

#### 통과 조건
- [ ] **외부 동작이 100% 동일** (입력→출력 같음)
- [ ] 모든 기존 테스트 통과
- [ ] 성능이 동일하거나 개선됨
- [ ] 관련 모든 엔드포인트 회귀 통과

## 🚨 위험 명령 실행 시 추가 기준

다음 명령을 사용한 경우, **사용자 명시 승인 + 백업 증거** 필수:

| 명령 | 추가 증거 |
|------|----------|
| `rm -rf` | 사용자 승인 채팅 인용 |
| `docker volume rm` | 사용자 승인 + 백업 명령 출력 |
| `DROP TABLE` | 사용자 승인 + DB 덤프 파일 경로 |
| `alembic downgrade` | 사용자 승인 + 현재 DB 상태 백업 |
| `git reset --hard` | 사용자 승인 + 현재 브랜치 백업 |

위 명령을 **승인 없이 실행했으면 무조건 실패** 처리.

## 📌 의심스러울 때

판단이 애매할 때는:

1. **사용자에게 묻는다.** 추측하지 않는다.
2. **실패로 분류한다.** 통과보다 실패가 안전하다.
3. **로그를 본다.** 코드보다 로그가 진실이다.

- [ ] reviewer가 APPROVED 판정함

- [ ] 모든 증거 (명령 출력, 스크린샷 등) 수집됨
- [ ] 보고서가 REPORT_TEMPLATE.md 양식대로 작성됨

하나라도 빠지면 보고하지 않는다.
