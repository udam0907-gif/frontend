# WORKFLOW.md — 일주일 개발 사이클 가이드

> 이 문서는 시니어 개발자처럼 작업하는 방법을 정리한다.
> 한 주를 어떻게 운영해야 안정적인 진척이 나오는지 보여준다.

---

## 🗓️ 한 주 사이클 (한눈에)

```
일요일 저녁 ─── 이번 주 목표 1개 결정 (30분)
      ↓
월요일 ─────── 스펙 작성 + 티켓 쪼개기 (1~2시간)
      ↓
화요일 ─────── 티켓 #1 (Claude Code + 워크플로우)
      ↓
수요일 ─────── 티켓 #2
      ↓
목요일 ─────── 티켓 #3 + 검증의 날
      ↓
금요일 ─────── git push + 회고 (30분)
      ↓
주말 ────────── 휴식 / 다음 주 영감
```

---

## 🎯 일요일 저녁: 목표 결정

### 원칙: 한 주에 큰 목표 1개만

❌ "PDF 업로드 + 참여연구원 + 422 오류 + UI 정리 다 끝낸다"
✅ **"PDF 업로드해서 폼 자동완성까지 안정적으로 작동하게 만들기"**

### 목표 SMART 체크
- **S**pecific: 구체적인가? (애매한 "개선" 금지)
- **M**easurable: 측정 가능한가? (어떻게 됐을 때 끝인가?)
- **A**chievable: 한 주에 가능한가?
- **R**elevant: 지금 이게 진짜 중요한가?
- **T**ime-bound: 금요일까지 끝낼 수 있는가?

### 작성 양식 (`이번주_목표.md`)

```markdown
## Week of YYYY-MM-DD

### 핵심 목표
[한 문장]

### 완료 정의
이게 되면 이번 주는 성공:
- [ ] [구체적 동작 1]
- [ ] [구체적 동작 2]
- [ ] [구체적 동작 3]

### 이번 주에 안 할 것
- [유혹 1]
- [유혹 2]

### 위험 요소
- [예상되는 막힐 지점]
```

## 📋 월요일: 티켓 쪼개기

### 원칙: 한 티켓 = 반나절~하루 분량

큰 목표를 작은 티켓으로 분해.

### 작성 양식 (`이번주_티켓.md`)

```markdown
## 티켓 목록

### #1 PDF 업로드 UI 보강
- 파일: `frontend/app/projects/new/page.tsx`
- 작업: 드래그앤드롭 영역 추가, 파일 선택 버튼 개선
- 완료 조건: 브라우저에서 PDF를 끌어다 놓으면 파일명이 표시됨
- 예상 시간: 3시간

### #2 422 오류 원인 진단
- 파일: `backend/app/api/v1/projects.py`
- 작업: extract-pdf 엔드포인트에서 422 발생 시 상세 로그 추가
- 완료 조건: 422 발생 시 어느 필드가 문제인지 응답에 포함
- 예상 시간: 2시간

### #3 폼 자동완성 연결
- 파일: `frontend/app/projects/new/page.tsx`, `frontend/lib/api.ts`
- 작업: extractPdf 응답을 폼 필드에 매핑
- 완료 조건: PDF 업로드 후 모든 필드가 채워지고 사용자가 수정 가능
- 예상 시간: 4시간
```

## 💻 화~목: 구현 (Claude Code 활용)

### 매 티켓마다 이 순서

#### 1. 작업 지시
```
티켓 #1 작업해줘.

[티켓 내용을 그대로 복사]

진행 절차:
1. explorer로 현재 코드 파악
2. 작업 계획 보고
3. 내가 승인하면 implementer로 수정
4. verifier로 검증
5. reviewer 통과 후 보고
```

#### 2. 작업 계획 받고 검토
Claude Code가 계획을 보고하면:
- 수정 파일이 티켓과 일치하는가?
- 영향 범위가 합리적인가?
- 보호 영역이 포함됐는가? (포함됐으면 거부)

승인하면 구현 시작.

#### 3. commit
```bash
git add .
git commit -m "feat: 티켓 #1 PDF 업로드 UI 보강"
```

### 하루 끝낼 때 체크
- [ ] 오늘 티켓 1개 완료했는가?
- [ ] git status 깨끗한가?
- [ ] 내일 티켓 준비됐는가?

## 🔍 목요일 저녁: 검증의 날

### 원칙: 새 코드 안 짜고 확인만

#### 1. 전체 회귀 테스트
```bash
# 과제 목록
curl -s http://localhost:8000/api/v1/projects/ | head -c 200

# 비목 목록
curl -s http://localhost:8000/api/v1/expenses/ | head -c 200

# RCMS Q&A
curl -X POST http://localhost:8000/api/v1/rcms/qa \
  -H "Content-Type: application/json" \
  -d '{"question":"테스트"}'
```

#### 2. UI 전체 둘러보기
모든 페이지를 직접 클릭해본다:
- 메인 대시보드
- 과제 등록
- 과제 상세
- 비용 등록
- 문서 생성
- RCMS Q&A

콘솔(F12)에 에러 없는지 확인.

## 📦 금요일: 정리

### 1. 전체 push
```bash
git push origin master
```

### 2. 회고 작성 (`회고_YYYY-WW.md`)

```markdown
## 이번 주 회고

### ✅ 완료한 것
- 티켓 #1: ___
- 티켓 #2: ___

### ❌ 못 끝낸 것
- 티켓 #3: 70% — 다음 주로 이월
- 사유: [무엇이 막혔나]

### 💡 배운 것 / 개선할 것
- AI 지시 시 [무엇]을 더 명확히 했어야 했다
- [원칙]이 잘 작동했다 / 안 작동했다

### 🎯 다음 주 후보
- [후보 1]
- [후보 2]
```

## 🚦 신호등 시스템

### 🟢 GREEN — 정상 진행
- 오늘 티켓 완료
- 검증 통과
- 새 버그 없음

### 🟡 YELLOW — 주의
- 오늘 티켓 미완료지만 진척 있음
- 사소한 사이드 이펙트 발견
- 시간 초과 30% 이내

### 🔴 RED — 위험
- 같은 문제로 4시간 이상 막힘
- 회귀 버그 발생
- 보호 영역 침범 발생

🔴 신호 시 액션:
1. **작업 중단**
2. 원인 분석
3. 사용자에게 도움 요청
4. 다른 접근으로 재시작 (시간 박스 정해서)
