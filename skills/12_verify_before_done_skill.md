# Skill: verify_before_done_skill

> "완료했습니다" 는 확인 후에만 말한다. 잘 되던 것이 갑자기 안 되면 안 된다.

---

## 규칙 1 — 수정 후 반드시 확인

코드를 수정했으면 반드시 해당 파일을 다시 읽어서 변경사항이 실제로 반영됐는지 확인한다.
확인 전에 "완료", "수정했습니다", "적용됐습니다" 라고 말하지 않는다.

```
수정 → 파일 재읽기 → 변경 확인 → 그 다음에 "완료"
```

---

## 규칙 2 — 실행으로 증명

가능한 작업은 반드시 실행 결과로 증명한다.

```bash
# 백엔드 수정 후
docker compose restart backend
docker compose logs backend --tail=20

# API 수정 후
curl -UseBasicParsing "http://localhost:8000/api/v1/..."

# DB 변경 후
docker compose exec postgres psql -U postgres -d rnd_expense_db -c "SELECT ..."
```

---

## 규칙 3 — 회귀 방지 (잘 되던 것이 갑자기 안 되면 안 된다)

파일을 수정할 때 반드시 아래를 확인한다:

1. **이 파일을 참조하는 다른 파일이 있는가?**
   - 있으면 해당 파일도 영향 받는지 확인
2. **기존에 동작하던 기능이 깨지지 않는가?**
   - 수정 범위 밖의 기능은 건드리지 않는다
3. **import, 함수명, 변수명을 바꿨으면?**
   - 참조하는 모든 곳을 같이 수정한다

---

## 규칙 4 — 수정 범위 최소화

요청받은 것만 수정한다. 요청하지 않은 코드는 건드리지 않는다.
"더 나은 코드"를 위해 관련 없는 부분을 같이 수정하지 않는다.

---

## 보고 형식 (작업 완료 시 반드시 이 형식으로)

```
수정 파일: (파일명)
변경 내용: (한 줄 요약)
확인 방법: (실행한 명령어 또는 재읽기 결과)
영향 범위: (다른 기능에 영향 없음 / 또는 영향 있는 파일 명시)
```

---

## 절대 금지

- 확인 없이 "완료했습니다" 말하기
- 수정 범위 밖 코드 변경
- import/함수명 변경 후 참조 파일 미확인
- 테스트 없이 "정상 동작합니다" 말하기
